# Правила для агента Kilo

Политика выровнена с корневыми правилами **`.cursor/rules`** (GRACE 3.x, история, git, среда, верификация без локального запуска приложения).

---

## Старт сессии

1. Прочитай **`docs/HISTORY.md`** (последние ~200 строк при большом файле). Если файла или каталога **`docs/`** нет — создай **`docs/HISTORY.md`** с кратким заголовком и веди журнал дальше.
2. Прочитай **`docs/AGENTS.md`** — единственная семантическая карта проекта при старте. Корневой **`AGENTS.md`** не использовать как замену. Если **`docs/AGENTS.md`** нет — создай минимальную структуру и поддерживай актуальность.
3. **XML и граф GRACE** — только в **`grace/**`**; не дублировать в **`docs/*.xml`**. Порядок чтения GRACE перед существенными правками:
   - `grace/knowledge-graph/knowledge-graph.xml`
   - `grace/plan/development-plan.xml`
   - `grace/verification/verification-plan.xml`
   - при фичах/сценариях — `grace/requirements/requirements.xml`
   - при стеке и семантике — `grace/technology/technology.xml`
4. Кратко подтверди пользователю восстановление контекста, затем приступай к задаче.

---

## История проекта (**`docs/HISTORY.md`**)

После **каждой** завершённой задачи дописывай: что сделано, почему, какие файлы затронуты, решения и компромиссы. Записывай **до коммита**. При изменениях GRACE указывай пути **`grace/**/*.xml`** и при необходимости **`grace/README.md`**.

---

## Git, версия и коммиты

- После задачи выполняй **`git add`**, **`git commit`**, **`git push`** (если доступно); не откладывай коммит инструкцией пользователю.
- **Единственный** источник номера версии — файл **`VERSION`** в корне (SemVer). Остальные манифесты в том же коммите приведи в соответствие с **`VERSION`**.
- **Subject коммита:** `X.Y.Z: краткое описание` (номер **в начале** строки, как в **`VERSION`**).

---

## GRACE Framework

- Данные GRACE хранятся **только** в **`grace/`**; см. **`grace/README.md`**.
- Перед правками читай **`MODULE_CONTRACT`** и маркеры в коде (`# GRACE`, `@grace` и т.д.); ориентир — **`grace/technology/technology.xml`**.
- **Обновляй `MODULE_MAP`** после изменения exports или вспомогательных поверхностей.
- **Синхронизация** (код и GRACE не должны расходиться):

| Что изменилось | Что обновить |
|----------------|--------------|
| Новый модуль / класс | `grace/knowledge-graph/knowledge-graph.xml`, `grace/plan/development-plan.xml` |
| Новый экспорт / функция | `grace/knowledge-graph/knowledge-graph.xml` |
| Новая зависимость между модулями | граф + при необходимости план |
| Маршруты, удаление модулей | граф, план, при необходимости `grace/verification/verification-plan.xml` |
| Логика проверки, тесты, маркеры | `grace/verification/verification-plan.xml` |
| Новый юзкейс | `grace/requirements/requirements.xml`, план, при необходимости верификация |
| Стек, тесты, логирование | `grace/technology/technology.xml`, при необходимости верификация |

- **Контракт-first:** не пиши код без актуального контракта; новый модуль — запись **`V-M-xxx`** в **`grace/verification/verification-plan.xml`**.
- После багфиксов добавляй **`CHANGE_SUMMARY`**; **не удаляй** семантическую разметку без необходимости.
- Существенные правки GRACE и разделов AGENTS фиксируй в **`docs/HISTORY.md`**.

---

## Среда и терминал

- Команды — для **Windows PowerShell** (или **`pwsh`**, если уместно). **Не** использовать **`bash`**, **`sh`**, **`cmd`** и Unix-стили там, где есть эквивалент PowerShell, **кроме** явной просьбы пользователя.
- Общение с пользователем — **на русском**, если не попросили иной язык.

---

## Запрет локального запуска приложения для проверки

**Не** поднимай и **не** прогоняй приложение (или его части) в терминале **ради проверки** в рантайме (Node/npm, PHP-серверы, Python dev-серверы, `docker compose up` для приложения и т.п.). Проверка: линтеры, статический анализ, чтение кода и контрактов. **Исключение:** пользователь **явно** просит конкретную команду запуска.
